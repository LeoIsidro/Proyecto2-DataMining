Count-Min Sketch

Estima la frecuencia de reseñas por business_id procesando el stream elemento a elemento. Matriz fija de contadores — sin tabla hash proporcional al número de elementos únicos.

Insights

100% sin subestimación
Ningún business_id tuvo error negativo en el experimento. La garantía f̂ ≥ f se cumple empíricamente.
Error real ≪ cota teórica
El error promedio es una fracción pequeña de ε·N — el sketch es mucho más preciso que lo garantizado.
w domina más que k
Doblar w reduce el error a la mitad. Doblar k reduce δ exponencialmente pero impacta menos el error promedio.

03 — Ventana Binaria Reciente

Algoritmo DGIM

¿Cuántas de las últimas N reseñas fueron "útiles"? — sin guardar los N bits del historial. Memoria O(log² N) en lugar de O(N).

Stream binario bₜ = 1 si useful > 0 Buckets: (timestamp, size) size = potencia de 2 Fusión cuando ≥ 3 del mismo tamaño Expiración al salir de la ventana N
Operaciones
1
Bit=1 → crear bucket (t, size=1). Bit=0 → no crear bucket.
2
Expirar buckets con timestamp < t − N.
3
Si ≥ 3 buckets de tamaño s → fusionar los 2 más viejos en 1 de tamaño 2s.
4
Consulta: Σ sizes − size_último / 2
Estimación f̂ = Σ(sizes) − ⌊size_último / 2⌋ suma de todos los buckets, menos la mitad del más antiguo Error máximo garantizado error < 50% en la práctica ~2% sobre reseñas Yelp Complejidad de actualización O(1) amortizado · O(log N) peor caso cascada de fusiones en el peor caso
MemoriaO(log² N)buckets activos
ActualizaciónO(1) amort.por bit recibido
Error máximo< 50%teórico
Ahorro vs exacto~300×para N=100,000

DGIM vs Count-Min Sketch — diferencias clave

Aspecto Count-Min Sketch DGIM
Pregunta ¿Cuántas veces aparece x en todo el stream? ¿Cuántos 1s en las últimas N posiciones?
Input Cualquier elemento (string, ID…) Stream binario {0, 1}
Memoria O(k·w) fija O(log² N)
Ventana Todo el stream acumulado Últimos N eventos solamente
Error ≤ ε·N (con prob. 1−δ) · nunca subestima < 50% · puede sobrestimar o subestimar
Ventaja principal Múltiples elementos simultáneos · error probabilístico acotado Olvida automáticamente eventos viejos · ideal para alertas recientes
Cuándo usarlo Ranking de frecuencias globales (top-k, heavy hitters) Monitoreo en tiempo real de comportamiento reciente

Insights

Error real ~2% vs cota 50%
El peor caso teórico rara vez ocurre. Sobre Yelp, DGIM es muy preciso.
~300× ahorro de memoria
Para N=100,000 bits: exacto usa 100 KB; DGIM usa ~330 bits.
Escalabilidad por negocio
k negocios con DGIM: O(k·log²N). Con exacto: O(k·N) — diferencia de órdenes de magnitud.

04 — Cierre

Conclusiones

Tres técnicas de streaming, una misma fuente de datos, tres preguntas distintas — todas con memoria acotada.

  • 1
    Ventanas deslizantes (1h · 4h · 1D) revelaron la estructura temporal completa del stream: tendencia de crecimiento, caída COVID-2020, pico nocturno 18–22h, estacionalidad mensual y posible concept drift — todo con acumuladores O(1) por ventana, sin almacenar ningún evento.
  • 2
    Count-Min Sketch (k=4, w=2000) estimó la frecuencia de miles de business_ids en 62.5 KB fijos, garantizando nunca subestimar. El error empírico está muy por debajo de ε·N. El trade-off memoria/error es controlable con w; la confianza con k.
  • 3
    DGIM respondió "¿cuántas reseñas útiles en las últimas N?" con O(log² N) memoria y ~2% de error real (cota 50%). A diferencia de CMS, olvida automáticamente eventos antiguos — ideal para monitoreo de comportamiento reciente por negocio a escala.
